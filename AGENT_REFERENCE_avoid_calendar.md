# AI 기반 익명 동선 충돌 회피 캘린더
## Codex 개발 에이전트 참고 문서

> 문서 목적  
> 이 문서는 Codex 또는 다른 코딩 에이전트가 프로젝트의 목적과 구현 범위를 이해하고, 별도의 장황한 설명 없이 개발을 시작할 수 있도록 작성된 기준 문서다.

# 1. 프로젝트 개요

## 1.1 프로젝트 한 줄 설명

등록된 사용자끼리 정확한 개인 일정은 공개하지 않으면서, 특정 날짜·시간·지역에서 마주칠 가능성을 익명으로 확인하고 AI가 대체 장소나 시간대를 추천하는 지도형 캘린더 웹 서비스다.

## 1.2 프로젝트 배경

회사 구성원은 평일에는 자주 만나지만, 휴일이나 개인 시간에는 우연히 마주치고 싶지 않을 수 있다.

예를 들어 사용자가 이번 주 일요일 오후에 영등포에서 시간을 보내고 싶을 때, 같은 서비스를 사용하는 다른 회사 구성원이 해당 시간대에 영등포 인근을 이미 등록했다면 지도에 충돌 가능 지역을 표시한다.

서비스는 다른 사람의 이름, 정확한 위치, 상세 일정을 공개하지 않는다. 사용자는 단지 해당 시간과 지역에 일정이 겹칠 가능성이 있다는 사실만 확인할 수 있다.

## 1.3 프로젝트 성격

- 사내 AI 경진대회 비개발 Track 제출 후보
- 분류: AI at Work & Life / 생활 향상
- 업무용 시스템이 아니라 일상생활의 불편을 줄이는 재미있는 AI 활용 사례
- 기술 시연보다 사용자가 문제를 쉽게 이해하고 체감할 수 있는 데모가 중요
- 기존에 제출한 로컬 문서 RAG 검색 프로젝트와는 별개의 프로젝트

# 2. 핵심 사용자 시나리오

## 시나리오 A: 자연어로 일정 등록

사용자가 다음과 같이 입력한다.

> 이번 주 일요일 오후 2시부터 6시까지 영등포에서 놀고 싶어.

AI가 문장을 분석하여 다음 구조로 변환한다.

```json
{
  "date": "2026-07-19",
  "startTime": "14:00",
  "endTime": "18:00",
  "locationName": "영등포",
  "radiusMeters": 1500
}
```

장소명은 지도 API 또는 지오코딩 API를 통해 위도와 경도로 변환한다.

## 시나리오 B: 기존 일정과 충돌

다른 사용자가 같은 날짜 오후 1시부터 5시까지 영등포역 반경 1.5km를 먼저 등록한 상태다.

현재 사용자가 입력한 일정은 시간과 공간이 모두 겹치므로 충돌로 판정한다.

서비스는 다음과 같이 표시한다.

> 선택한 시간대에 영등포 인근에서 다른 일정과 겹칠 가능성이 높습니다.

중요한 점은 다른 사용자의 이름이나 정확한 장소를 공개하지 않는 것이다.

## 시나리오 C: AI 대안 추천

충돌이 발생하면 AI가 시스템에서 계산한 후보를 바탕으로 설명한다.

예시:

> 영등포는 오후 2시부터 5시까지 충돌 가능성이 높습니다.  
> 가까운 대안으로 신도림 또는 당산을 추천합니다.  
> 영등포를 유지하려면 오후 7시 이후가 상대적으로 안전합니다.

실제 거리 계산, 시간 중복 계산, 후보 탐색은 일반 코드로 수행한다. AI는 계산 결과를 자연스럽게 설명하고 사용자의 조건을 해석하는 역할만 담당한다.

# 3. 프로젝트 목표

## 3.1 최우선 목표

다음 데모 흐름이 처음부터 끝까지 정상적으로 동작해야 한다.

1. 사용자가 날짜를 선택한다.
2. 자연어 또는 직접 입력으로 시간과 장소를 등록한다.
3. AI가 자연어 입력을 구조화한다.
4. 지도에 선택 장소와 충돌 가능 영역이 표시된다.
5. 서버가 기존 일정과의 시간·거리 충돌을 계산한다.
6. 충돌하면 AI가 대체 지역 또는 시간대를 추천한다.
7. 충돌하지 않으면 일정을 저장한다.
8. 다른 사용자의 신원과 정확한 위치는 노출하지 않는다.

## 3.2 비목표

초기 MVP에서는 다음 기능을 구현하지 않는다.

- 실시간 GPS 위치 추적
- 백그라운드 위치 수집
- 회사 인사 시스템 연동
- 사내 계정 SSO
- 복잡한 권한 체계
- 모바일 네이티브 앱
- 실제 길찾기 기반 이동 시간 계산
- 대규모 사용자 트래픽 대응
- 결제 또는 구독 기능
- 관리자용 직원 위치 조회
- 다른 사용자의 상세 일정 열람
- AI 모델 자체 학습

# 4. 서비스 표현 원칙

서비스를 “회사 사람을 못 가게 막는 앱”으로 표현하지 않는다.

권장 표현:

> 정확한 개인 일정을 공개하지 않고, 등록된 구성원 간의 일정·지역 중복 가능성을 확인하며, AI가 대체 장소와 시간을 추천하는 익명 동선 조정 서비스

서비스 안에서는 다음 용어를 사용한다.

- 선점: 가능하면 사용하지 않거나 보조적으로만 사용
- 금지 지역: 사용하지 않음
- 충돌 가능 지역: 사용
- 겹침 가능성: 사용
- 대체 장소 추천: 사용
- 익명 일정: 사용
- 프라이버시 보호: 사용

# 5. MVP 기능 범위

## 5.1 필수 기능

### 사용자

- 간단한 닉네임 로그인
- 데모에서는 비밀번호 없는 임시 사용자 방식도 허용
- 사용자별 자신의 일정 조회
- 자신의 일정 삭제

### 캘린더

- 월간 캘린더 표시
- 날짜 선택
- 선택 날짜의 일정 상태 표시
- 오늘 날짜 강조

### 지도

- 선택한 장소 중심으로 지도 이동
- 사용자가 선택한 위치에 마커 표시
- 충돌 가능 영역을 반투명한 빨간 원으로 표시
- 선택 가능한 본인 후보 영역은 일반 원 또는 다른 형태로 표시
- 다른 사람의 정확한 마커는 표시하지 않음

### 일정 등록

두 가지 입력 방식을 지원한다.

1. 자연어 입력
2. 직접 입력

직접 입력 필드:

- 날짜
- 시작 시간
- 종료 시간
- 장소명
- 위도
- 경도
- 충돌 확인 반경

### 충돌 판정

다음 조건이 모두 충족되면 충돌로 판정한다.

1. 날짜가 같다.
2. 시간 구간이 겹친다.
3. 두 일정의 중심 좌표 간 거리가 각 일정의 반경 합보다 작거나 같다.

### AI 기능

- 자연어 일정 문장 구조화
- 애매한 시간 표현 해석
- 충돌 결과 설명
- 대체 시간 또는 지역 추천 문장 생성

### 프라이버시

- 다른 사용자의 이름 미표시
- 다른 사용자의 정확한 좌표 미표시
- 다른 사용자의 상세 시간 미표시
- 지도에는 익명화된 충돌 범위만 표시
- API 응답에도 불필요한 사용자 식별 정보를 포함하지 않음

## 5.2 선택 기능

시간이 남으면 아래 기능을 추가한다.

- 사용자가 원하는 조건 입력
  - 지하철로 너무 멀지 않은 곳
  - 영화관이 있는 지역
  - 카페가 많은 지역
- 위험도 점수
- 추천 장소 원클릭 적용
- 추천 시간 원클릭 적용
- 일정 수정
- 데모 사용자 전환
- 날짜별 충돌 개수 표시
- 다크 모드
- 간단한 온보딩

# 6. 권장 기술 스택

## 6.1 기본 권장안

- 프론트엔드: Next.js
- 언어: TypeScript
- UI: Tailwind CSS
- 캘린더: 직접 구현 또는 React 기반 캘린더 라이브러리
- 지도: Kakao Maps JavaScript API
- 백엔드: Next.js Route Handler 또는 Server Action
- 데이터베이스: SQLite + Prisma
- AI: OpenAI 호환 API 또는 로컬 Ollama
- 검증: Zod
- 날짜 처리: date-fns
- 테스트: Vitest
- E2E 선택: Playwright

## 6.2 이 구성을 권장하는 이유

- 하나의 저장소에서 화면과 API를 함께 관리 가능
- SQLite를 사용하면 별도의 DB 서버 없이 로컬 데모 가능
- Prisma로 데이터 구조와 마이그레이션 관리 가능
- TypeScript로 프론트와 서버 타입 공유 가능
- 경진대회 데모용으로 설치와 실행이 비교적 단순함

## 6.3 지도 API 대체

Kakao Maps API 설정이 어려우면 다음 순서로 대체 가능하다.

1. Kakao Maps
2. Naver Maps
3. OpenStreetMap + Leaflet

에이전트는 지도 API 키가 없어도 개발이 가능하도록 지도 컴포넌트를 분리하고, 키가 없을 경우 임시 좌표 기반 목업 모드를 제공해야 한다.

# 8. 데이터 모델

## 8.1 User

```prisma
model User {
  id        String     @id @default(cuid())
  nickname  String
  createdAt DateTime   @default(now())
  schedules Schedule[]
}
```

## 8.2 Schedule

```prisma
model Schedule {
  id             String   @id @default(cuid())
  userId         String
  date           DateTime
  startMinutes   Int
  endMinutes     Int
  locationName   String
  latitude       Float
  longitude      Float
  radiusMeters   Int      @default(1500)
  createdAt      DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([date])
  @@index([userId])
}
```

## 8.3 시간 저장 방식

시간은 `HH:mm` 문자열 대신 자정 기준 분 단위 정수로 저장한다.

예시:

```text
14:00 → 840
18:30 → 1110
```

장점:

- 시간 구간 비교가 간단함
- 타임존 문제를 줄일 수 있음
- MVP에서는 하루를 넘는 일정을 허용하지 않음

## 8.4 추천 후보

추천 후보는 초기에는 DB에 저장하지 않고 서버에서 임시 생성한다.

```ts
type RecommendationCandidate = {
  type: "location" | "time";
  title: string;
  reason: string;
  latitude?: number;
  longitude?: number;
  startMinutes?: number;
  endMinutes?: number;
  estimatedRisk: "low" | "medium" | "high";
};
```

# 9. 충돌 계산 규칙

## 9.1 시간 중복

두 일정 A와 B가 있을 때:

```ts
const overlaps =
  a.startMinutes < b.endMinutes &&
  b.startMinutes < a.endMinutes;
```

다음은 충돌하지 않는다.

```text
A: 10:00~12:00
B: 12:00~14:00
```

다음은 충돌한다.

```text
A: 10:00~12:30
B: 12:00~14:00
```

## 9.2 거리 계산

두 위도·경도 사이의 거리는 Haversine 공식을 사용한다.

반환 단위는 미터다.

## 9.3 공간 중복

```ts
const spatialOverlap =
  distanceMeters <= a.radiusMeters + b.radiusMeters;
```

## 9.4 최종 충돌

```ts
const isConflict =
  sameDate &&
  timeOverlap &&
  spatialOverlap;
```

## 9.5 위험도 점수

위험도 점수는 선택 기능이지만 구현한다면 규칙 기반으로 계산한다.

예시:

```text
시간 중복률: 0~50점
거리 근접도: 0~40점
동일 시간대 익명 일정 수: 0~10점
총점: 0~100점
```

AI가 점수를 직접 만들어서는 안 된다.

# 10. AI 역할

## 10.1 AI가 담당할 작업

- 자연어에서 날짜 추출
- 자연어에서 시작·종료 시간 추출
- 자연어에서 장소명 추출
- “점심쯤”, “저녁 전”, “오후 내내” 같은 표현 해석
- 사용자의 추가 조건 추출
- 계산된 충돌 결과를 자연어로 설명
- 시스템이 생성한 후보를 바탕으로 추천 문장 작성

## 10.2 AI가 담당하지 않을 작업

- 거리 계산
- 시간 중복 계산
- 충돌 최종 판정
- DB 직접 수정
- 임의의 사용자 정보 조회
- 정확한 지도 좌표 생성
- 존재하지 않는 장소를 확정적으로 추천
- 다른 사용자의 신원 추론

## 10.3 핵심 원칙

> 계산은 코드가 하고, AI는 문장을 이해하고 결과를 설명한다.

# 11. AI 자연어 일정 파싱

## 11.1 입력 예시

```text
이번 주 일요일 오후 2시부터 저녁 먹기 전까지 영등포에서 놀고 싶어.
```

## 11.2 출력 스키마

```ts
const ParsedScheduleSchema = z.object({
  date: z.string().date(),
  startTime: z.string().regex(/^\d{2}:\d{2}$/),
  endTime: z.string().regex(/^\d{2}:\d{2}$/),
  locationName: z.string().min(1),
  radiusMeters: z.number().int().min(100).max(10000).default(1500),
  preferences: z.object({
    maxTravelMinutes: z.number().int().positive().nullable(),
    needsCinema: z.boolean().nullable(),
    needsCafe: z.boolean().nullable()
  }).default({
    maxTravelMinutes: null,
    needsCinema: null,
    needsCafe: null
  }),
  assumptions: z.array(z.string()).default([])
});
```

## 11.3 AI 시스템 프롬프트 방향

```text
당신은 한국어 일정 문장을 구조화하는 도우미다.

반드시 지정된 JSON 형식으로만 답한다.
오늘 날짜와 사용자 타임존은 서버가 제공한다.
날짜, 시간, 장소가 불명확하면 합리적인 기본값을 적용하되 assumptions에 기록한다.
다른 사람의 일정이나 개인정보는 추론하지 않는다.
```

## 11.4 애매한 표현 기본값

초기 MVP에서는 다음 기본값을 사용한다.

- 아침: 09:00~12:00
- 점심: 12:00~14:00
- 오후: 13:00~18:00
- 저녁: 18:00~21:00
- 밤: 21:00~23:00
- 하루 종일: 10:00~20:00
- 저녁 전까지: 종료 18:00
- 점심부터 저녁 전까지: 12:00~18:00

AI 응답의 `assumptions`에 적용한 기준을 기록하고 UI에서 사용자에게 보여준다.

# 12. AI 추천 생성

## 12.1 추천 생성 순서

1. 서버가 충돌 일정을 조회한다.

3. 서버가 대체 시간 후보를 생성한다.

## 15.2 일정 목록

```text

```

일반 사용자 응답에는 다른 사용자의 상세 일정이 포함되어서는 안 된다.

본인 일정과 익명화된 충돌 영역을 분리해서 반환한다.

## 15.3 충돌 검사

```text
POST /api/schedules/conflicts
```

요청:

```json
{
  "date": "2026-07-19",
  "startMinutes": 840,
  "endMinutes": 1080,
  "latitude": 37.5159,
  "longitude": 126.9075,
  "radiusMeters": 1500
}
```

응답:

```json
{
  "hasConflict": true,
  "riskScore": 82,
  "anonymousConflictCount": 2,
  "overlapWindow": {
    "startMinutes": 840,
    "endMinutes": 1020
  },
  "displayAreas": [
    {
      "latitude": 37.5165,
      "longitude": 126.9081,
      "radiusMeters": 1800
    }
  ]
}
```

`displayAreas`는 개인 일정 하나하나를 그대로 반환하지 않고, 필요하면 여러 일정을 합치거나 좌표를 둔화시켜 익명화한다.


서버는 저장 직전에 충돌을 다시 검사해야 한다.

## 16.1 금지 사항

- 실시간 위치 수집 금지

## 16.2 지도 표시 익명화

초기 구현에서는 다음 중 하나를 선택한다.

### 방식 A: 좌표 둔화

다른 일정 좌표를 일정한 격자 단위로 반올림한다.

### 방식 B: 충돌 영역 합치기

서로 겹치는 원들을 하나의 큰 충돌 영역으로 표시한다.

### 방식 C: 중심점 미표시

정확한 중심 마커 없이 반투명 영역만 표시한다.

MVP 권장 방식은 C다.

## 16.3 데이터 삭제

데모에서는 지난 일정 데이터를 자동 삭제하지 않아도 되지만, 확장 계획에는 다음 내용을 명시한다.

- 일정 종료 후 일정 기간이 지나면 자동 삭제
- 사용자가 자신의 일정 즉시 삭제 가능
- 위치 데이터 최소 수집

# 17. 예외 처리

다음 상황을 처리해야 한다.

- 종료 시간이 시작 시간보다 빠른 경우
- 날짜가 과거인 경우
- 장소를 찾지 못한 경우
- 지도 API 키가 없는 경우
- AI 응답이 JSON 스키마와 맞지 않는 경우
- AI 서버에 연결할 수 없는 경우
- 동일 사용자가 동일 일정을 중복 등록한 경우
- 저장 직전에 다른 사용자가 먼저 같은 지역을 등록한 경우
- DB가 비어 있는 경우
- AI 없이 직접 입력만으로도 일정 등록이 가능해야 함

AI 기능이 실패해도 전체 서비스가 멈추지 않도록 한다.

# 18. 동시성 처리

두 사용자가 동시에 같은 지역과 시간을 등록할 수 있다.

따라서 다음 순서로 처리한다.

1. 사용자가 등록 요청
2. 서버에서 최신 데이터 기준 충돌 재검사
3. 충돌이 없으면 트랜잭션 안에서 저장
4. 저장 직전 또는 트랜잭션 내부에서 다시 조건 확인
5. 충돌이 새로 발생했다면 등록 거절

SQLite MVP에서는 완전한 공간 제약 조건을 DB에 넣기 어렵기 때문에 서버 트랜잭션과 짧은 처리 구간으로 대응한다.

# 20. 테스트 요구사항

## 20.1 필수 단위 테스트

- 시간 구간이 겹치는 경우
- 시간 구간이 경계에서 닿기만 하는 경우
- 같은 날짜지만 시간이 다른 경우
- 시간이 같지만 거리가 먼 경우
- 시간과 거리가 모두 겹치는 경우
- Haversine 거리 계산
- 잘못된 AI JSON 거절
- 시작 시간이 종료 시간보다 늦은 경우 거절

## 20.2 통합 테스트

- 자연어 입력 → 구조화 결과 표시
- 구조화 결과 → 충돌 검사
- 충돌 없음 → 일정 등록
- 충돌 있음 → 추천 표시
- AI 실패 → 직접 입력 사용 가능
- 다른 사용자의 개인정보가 API 응답에 없는지 확인

# 23. 에이전트 작업 원칙

Codex는 다음 원칙을 지킨다.

1. 한 번에 전체를 무리하게 구현하지 않고 단계별로 진행한다.
2. 각 단계가 끝날 때 실행 가능한 상태를 유지한다.
3. 핵심 계산을 AI에게 맡기지 않는다.
4. AI 응답은 반드시 스키마 검증 후 사용한다.
5. 다른 사용자의 개인정보를 API 응답에 포함하지 않는다.
6. 복잡한 기능보다 데모 흐름의 안정성을 우선한다.
7. 지도 API 키가 없어도 목업 모드로 실행 가능하게 한다.
8. 모든 주요 설정은 `.env`로 분리한다.
9. 에러가 나면 임시로 숨기지 말고 원인과 해결 방법을 기록한다.
10. 구현 변경 시 README와 이 문서를 함께 갱신한다.

# 25. 발표용 5분 데모 시나리오

## 장면 1: 문제 제시

> 쉬는 날 영등포에서 놀려고 했는데 회사 사람과 우연히 마주칠까 신경 쓰인 적이 있습니다.

## 장면 2: 기존 익명 일정

사용자 A가 일요일 오후 영등포를 등록한 상태를 준비한다.

## 장면 3: 자연어 입력

사용자 B가 입력한다.

> 이번 주 일요일 오후 2시부터 6시까지 영등포에서 영화 보고 싶어.

AI가 날짜, 시간, 장소를 자동 분석한다.

## 장면 4: 충돌 표시

지도에 영등포 인근 충돌 가능 영역이 빨간색으로 표시된다.

화면에는 다음만 표시한다.

- 익명 일정 1개 이상
- 충돌 가능 시간
- 위험도

## 장면 5: AI 추천

AI가 추천한다.

- 신도림
- 당산
- 영등포 오후 7시 이후

## 장면 6: 대안 선택

신도림을 선택하고 일정을 등록한다.

## 장면 7: 마무리

> 정확한 위치나 신원을 공개하지 않고도 원치 않는 일정 중복을 피할 수 있습니다.

# 26. 경진대회 제출 문구 초안

## AI를 어디에 활용했나요?

휴일이나 개인 일정에서 회사 구성원과 원치 않게 마주칠 가능성을 줄이기 위한 익명 지도형 캘린더에 AI를 활용했다.

## AI를 어떻게 활용했나요?

사용자가 입력한 “이번 주 일요일 오후에 영등포에서 놀고 싶다”와 같은 자연어를 날짜, 시간, 장소 정보로 변환했다. 시스템이 시간과 거리 기반으로 익명 일정 충돌을 계산한 뒤, AI가 충돌 원인을 설명하고 겹치지 않는 인근 지역이나 다른 시간대를 추천하도록 구성했다.

## AI를 활용해서 무엇이 좋아졌나요?

사용자가 복잡한 날짜·시간·장소 폼을 직접 입력하지 않아도 되었고, 단순히 충돌 여부만 확인하는 것이 아니라 상황에 맞는 대체 장소와 시간을 자연어로 추천받을 수 있게 되었다. 또한 다른 사용자의 정확한 일정과 위치를 공개하지 않아도 익명으로 동선 중복 가능성을 확인할 수 있다.

# 27. Codex 최초 실행 지시문

아래 지시문을 Codex에 함께 전달한다.

```text
현재 폴더에 있는 AGENT_REFERENCE.md를 프로젝트의 최우선 요구사항 문서로 사용해라.

먼저 문서를 읽고 다음 작업만 수행해라.

1. Next.js, TypeScript, Tailwind, Prisma, SQLite 기반 프로젝트를 초기화한다.
2. 권장 폴더 구조를 만든다.
3. User와 Schedule Prisma 모델을 작성한다.
4. 데모 시드 데이터를 작성한다.
5. 시간 중복, 거리 계산, 최종 충돌 판정 함수를 구현한다.
6. 해당 계산 함수의 단위 테스트를 작성한다.
7. 기본 메인 화면과 월간 캘린더의 뼈대를 만든다.
8. README에 설치 및 실행 방법을 작성한다.

아직 지도 API와 AI 호출은 구현하지 마라.
1단계 구현이 끝난 뒤 변경 파일, 실행 방법, 테스트 결과, 다음 단계 작업을 정리해서 보고해라.

중요 원칙:
- 계산 로직과 UI를 분리한다.
- TypeScript strict 모드를 유지한다.
- 핵심 함수에 타입을 명시한다.
- 오류를 임시로 숨기지 않는다.
- npm run dev와 npm test가 실행 가능한 상태로 마무리한다.
```

# 29. 현재 구현 확장 상태

- MVP 1~8단계의 핵심 데모 흐름 구현 완료
- 로컬 Ollama 모델은 `AI_MODEL` 환경변수로 교체 가능
- 충돌 없는 후보를 일반 코드로 검증한 뒤 Ollama가 설명하도록 구현
- HttpOnly 쿠키 기반 로컬 데모 세션 구현
- 초대 코드 기반 비공개 그룹 생성·참여 구현
- 충돌 비교 범위를 같은 비공개 그룹 구성원으로 제한
- 익명 충돌 집계와 추천 후보 계산에서 현재 사용자의 일정 제외

로컬 데모 세션은 인증 흐름과 서버 측 소유권 검사를 보여주기 위한 기능이며, 실제 운영 인증을 대체하지 않는다. 배포 시에는 배포 환경이 제공하는 인증 또는 사내 인증을 별도로 연결한다.
